## What changed in this session

### 1. Holistic judge (`tuning/gepa/judge.py`)
The judge now sees the **full prediction** (5 module outputs: flags,
response_text, field_ids, field_values, question, options,
summary_title, summary_content) PLUS the full case input. Previously it
saw only `response_text`. Rubric items can now reason about cross-module
consistency (e.g., "did response_text claim an action that field_ids
didn't perform?").

`judge_case(case, predicted: dict, rubric_items)` — note `predicted` is
a dict, not a string.

### 2. Two-layer rubric (`tuning/gepa/rubrics.py`)
Replaced 28 per-seed rubrics with:

- **UNIVERSAL_GOOD** (5 items, applied to every case):
  `intent_acknowledged` (1.5), `internally_consistent` (2.0),
  `grounded_in_input` (2.0), `appropriate_tone` (0.5),
  `correct_modules` (1.5)
- **CATEGORY_BAD** (one item per CANNOT tag, applied only when that tag
  is in `case["cannot_targets"]`, weight 2.5 each):
  `#1_vibe`/`honors_social_signal`,
  `#2_text_action_disconnect`/`text_matches_action`,
  `#3_state_check`/`accurate_state_report`,
  `#4_commits_regardless`/`no_unprompted_commit`,
  `#5_group_indices`/`correct_group_target`,
  `#7_wrong_field_value`/`correct_field_value_pairing`,
  `#8_training_data_hallucination`/`no_training_data_leak`

`get_rubric_for_case(case)` = UNIVERSAL_GOOD + union of CATEGORY_BAD
items for each tag. Most cases get 6 items; dual-tag seeds get 7.

### 3. Tighter composite metric (`tuning/gepa/metric.py`)
Four changes to stop inflating scores via trivially-passing checks:
- **A.** `no_persona_leak` weight 3.0 → 0.5 (passes on ~90% of cases trivially)
- **B.** `appropriate_tone` weight 1.0 → 0.5 (judge passes this on nearly everything)
- **C.** `action_router` now uses **F1 over True flags** (not 5-flag
  exact match), AND is **skipped entirely** when both gold and pred have
  all-False flags. Mirrors the data_extractor "active only" pattern.
- **D.** Removed active `choice_builder` and `review_builder` structure
  checks (they only checked "is the output non-empty"; rubric's
  `internally_consistent` covers cross-module coherence). Kept inactive
  scorers (spurious-fire traps).

Measured deflation on 4 representative seeds:

| Seed | Old | New | Δ |
|---|---|---|---|
| P2-ex5 (text-action) | 0.499 | 0.368 | −0.131 |
| P12-ex2 (state) | 0.620 | 0.381 | −0.239 |
| P3-ex3 (commits) | 0.864 | 1.000 | +0.136* |
| R5 (text-action) | 0.517 | 0.412 | −0.105 |

*P3-ex3 ↑: the model's response is "decent" — doesn't commit data, just
imposes a format opinion. All rubric items legitimately pass under the
new metric. If we want this to fail, that's a rubric tightening
question.

## Open issues / next steps

1. **Plateau at +0.011.** GEPA found one good candidate (iter 15)
   then couldn't beat it for ~15 iterations on the GPT-5 judge. Could
   be: budget too small for this metric difficulty, GPT-5's
   inconsistency (it's the noisiest judge per the 5-judge test), or
   genuine local optimum.

2. **choice_builder can't be optimized by GEPA reflective mutation.**
   Change D removed scoring for the active path, so GEPA has no
   examples to learn from for that module. If choice_builder text
   quality matters, need a different approach: either re-introduce a
   minimal active scorer (e.g., enum-fidelity check) OR add a
   choice_builder-specific rubric item that fires on `#needs_choice`
   cases.

3. **Judge cost.** GPT-5 run was projected to cost ~$30-50 for full
   4000 rollouts; we halted at 48% so probably ~$15-25 actual. Judge
   cache (proposed but **not implemented**) would have saved 30-50%
   on retry-heavy workloads. Sketch in conversation history if anyone
   wants to wire it up — ~25 lines in `judge.py`, key on
   `(test_id, hash(pred), rubric asks, model)`.

4. **28-seed comparison in progress.** Background task `bzi88gky7`
   running `/tmp/seed_diff.py --optimized
   tuning/gepa/results/run_full_gpt5_20260510_2344/best_program.json
   --judge`. If still running, let it finish; if killed, restart it.

5. **P3-ex3-style "decent but opinionated" responses scoring 1.0.**
   The new rubric is lenient on the model imposing a format choice
   when the user asked a clarification question. If we want to penalize
   that, tighten `intent_acknowledged` or add a new universal item.

6. **Memory rules learned in this session**:
   - Match answer length to question length; verify before guessing
   - User override beats my recommendation (don't substitute back)
   - Default to minimal scope, no scaffolding ("stop" is a hard interrupt)
